# 26 — Release Policy

> **Status:** Official
> **Scope:** كل إصدار يخرج من مشروع Void — من أصغر patch إلى أكبر major release — بلا استثناء
> **Authority:** هذا الملف هو المرجع الرسمي الوحيد الذي يحكم عملية إصدار Void. لا يُطلَق أي إصدار دون اتباع ما هو مكتوب هنا. لا استثناءات بسبب ضغط الوقت، ولا تجاوزات بسبب "الإصدار صغير". كل إصدار — مهما كان حجمه — يمر بنفس الصرامة.

## 1. Release Philosophy

### 1.1 الإصدار تعهّد — ليس نقطة في التاريخ

كل إصدار من Void هو تعهّد للمستخدمين الذين يعتمدون عليه. التعهّد يعني: هذا الكود مستقر، اختُبر، وسيعمل كما هو موعود. إصدار يُطلَق بدون تحقق كافٍ ليس إصداراً — هو تجربة تُجرى على المستخدمين.

### 1.2 الاستقرار يسبق السرعة دائماً

الضغط لإطلاق إصدار بسرعة هو أكثر الأسباب شيوعاً لإصدار معطوب. Void يتبنى مبدأً واحداً: **إصدار واحد صحيح خير من ثلاثة إصدارات تصحيحية**. الوقت المُنفَق على التحقق قبل الإصدار يوفّر وقتاً أضعافه في إصلاح ما بعده.

### 1.3 كل إصدار قابل للتراجع

لا يُطلَق إصدار ما لم تكن استراتيجية التراجع عنه واضحة ومُختبَرة قبل الإطلاق. إصدار بلا خطة تراجع هو إصدار يضع المشروع في موقف لا عودة منه إذا فشل.

### 1.4 الشفافية مع المستخدمين

كل إصدار يصاحبه Changelog واضح يصف:
- ما الذي تغيّر
- ما الذي أُصلح
- ما الذي كُسر بشكل متعمّد (Breaking Changes)
- ما الذي يتطلب إجراءً من المستخدم (Migrations)

المستخدم يستحق معرفة ما سيواجهه قبل أن يُحدِّث.

### 1.5 لا "إصدار للإنتاج بدون اختبار"

هذه الجملة محظورة في Void. كل شيء يذهب للإنتاج يمر بـ QA. البيئة التي يُختبَر فيها الكود ليست الإنتاج — هي بيئة Staging مخصصة. الإنتاج ليس ساحة اختبار.

## 2. Versioning Strategy

### 2.1 رقم الإصدار — البنية الأساسية

يتبع Void نظام **Semantic Versioning 2.0.0** (SemVer) بشكل صارم:

```
MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]

أمثلة:
  1.0.0           ← إصدار مستقر
  1.2.3           ← MAJOR=1, MINOR=2, PATCH=3
  2.0.0-alpha.1   ← Pre-release للإصدار الكبير القادم
  1.3.0-beta.2    ← Beta لميزة جديدة
  1.2.4+20260703  ← Patch مع Build metadata
```

### 2.2 أنواع الإصدارات

| النوع | متى يُطلَق | دورة الاختبار |
|---|---|---|
| **MAJOR** | تغييرات جذرية تكسر التوافق | اختبار كامل — أسابيع |
| **MINOR** | ميزات جديدة متوافقة مع ما قبلها | اختبار موسّع — أيام |
| **PATCH** | إصلاحات أخطاء متوافقة | اختبار مركّز — ساعات |
| **Pre-release** | إصدار تجريبي للاختبار الخارجي | لا ضمانات استقرار |
| **Hotfix** | إصلاح عاجل لثغرة أمنية أو انهيار | مسار طارئ مُسرَّع |

### 2.3 Pre-release Labels

```
alpha   ← غير مستقر، للتطوير الداخلي فقط
beta    ← أكثر استقراراً، للاختبار الخارجي المحدود
rc      ← Release Candidate — مرشح للإصدار الرسمي
```

**الترتيب:** `1.0.0-alpha.1` < `1.0.0-alpha.2` < `1.0.0-beta.1` < `1.0.0-rc.1` < `1.0.0`

### 2.4 سياسة الفروع

```
main        ← دائماً يعكس آخر إصدار مستقر
develop     ← تجميع التطوير الجاري
release/*   ← تحضير الإصدارات (release/1.3.0)
hotfix/*    ← إصلاحات عاجلة (hotfix/1.2.5)
feature/*   ← ميزات جديدة
```

لا يُدمج مباشرة في `main` إلا من فرع `release/*` أو `hotfix/*`.

## 3. Semantic Versioning

### 3.1 متى يرتفع PATCH (x.y.Z)

يرتفع PATCH عند كل تغيير **متوافق مع ما قبله** ويُصلح سلوكاً خاطئاً:

```
✅ إصلاح خطأ في منطق Command Handler
✅ إصلاح Memory Leak في Session Manager
✅ إصلاح Crash عند استقبال payload غير متوقع
✅ إصلاح Race Condition في Event Dispatcher
✅ تحسين رسالة خطأ دون تغيير نوعها
✅ تحديث Dependency لإصلاح ثغرة أمنية (patch-level)
```

**القاعدة:** إذا كان الكود الذي يستخدم هذا الإصدار سيعمل بدون تعديل — هذا PATCH.

### 3.2 متى يرتفع MINOR (x.Y.z)

يرتفع MINOR عند إضافة **قدرة جديدة متوافقة مع ما قبلها**:

```
✅ إضافة Command جديد للبوت
✅ إضافة Plugin جديدة
✅ إضافة حقل اختياري جديد في API Response
✅ إضافة Event Type جديد في الكتالوج
✅ إضافة Method جديدة لـ Service قائمة
✅ تحسين أداء بدون تغيير السلوك الخارجي
```

**القاعدة:** إذا أضفت شيئاً جديداً ولم تكسر شيئاً قائماً — هذا MINOR.

### 3.3 متى يرتفع MAJOR (X.y.z)

يرتفع MAJOR عند أي تغيير **يكسر التوافق مع ما قبله** (Breaking Change):

```
✅ تغيير Interface في Core (حذف Method أو تغيير Signature)
✅ تغيير بنية Database Schema بطريقة تكسر الإصدار القديم
✅ إعادة تسمية Event Types قائمة في الكتالوج
✅ تغيير سلوك Session بطريقة تكسر Plugins قائمة
✅ حذف Command قائم أو تغيير Pattern بشكل جذري
✅ تغيير أسلوب Authentication بالكامل
```

**القاعدة:** إذا يحتاج أي شيء يعتمد على هذا الكود تعديلاً ليتوافق — هذا MAJOR.

### 3.4 PATCH يُعيد MINOR و MINOR يُعيد PATCH

عند رفع MAJOR، تُعاد MINOR وPATCH إلى صفر: `1.9.8` → `2.0.0`
عند رفع MINOR، تُعاد PATCH إلى صفر: `1.9.8` → `1.10.0`
عند رفع PATCH: `1.9.8` → `1.9.9`

### 3.5 الإصدار 0.y.z — مرحلة ما قبل الاستقرار

كل شيء بـ MAJOR = 0 يُعدّ في طور التطوير. API قابل للتغيير في أي وقت. لا ضمانات توافق بين الإصدارات في هذه المرحلة. التحول إلى `1.0.0` يعني إعلان الاستقرار والالتزام بـ SemVer بشكل كامل.

## 4. Compatibility Rules

### 4.1 تعريف التوافق في Void

التوافق في Void يعني: **أي Plugin أو Command أو Service بُني مقابل الإصدار X.y.z يعمل دون تعديل مع X.y+1.z و X.y.z+1**.

هذا يشمل:
- Core Interfaces لم تتغير بشكل يكسر المُطبِّقين
- Event Types القائمة لم تُحذف أو تُغيَّر بنيتها
- Session Format متوافق مع الجلسات المحفوظة
- Database Schema قابل للترقية دون فقدان بيانات

### 4.2 ما لا يُعدّ Breaking Change

```
✅ إضافة حقل اختياري جديد لـ Event Payload
✅ إضافة Method جديدة لـ Interface (اختياري مع default implementation)
✅ إضافة Event Type جديد في الكتالوج
✅ تحسين رسائل الأخطاء نصياً دون تغيير أنواعها
✅ إضافة Index جديد لقاعدة البيانات
✅ تقليل وقت استجابة Operation
✅ إضافة Logging إضافي
```

### 4.3 ما يُعدّ Breaking Change دائماً

```
❌ حذف Method من Interface
❌ تغيير نوع Parameter في Method (حتى لو "متوافق" منطقياً)
❌ جعل حقل اختياري إلزامياً
❌ تغيير بنية Event Payload بحذف حقل
❌ إعادة تسمية Event Type
❌ تغيير قيمة Enum موجودة
❌ تغيير سلوك Method القائم بطريقة يعتمد عليها الكود
❌ تغيير ترتيب Parameters
❌ تغيير نوع Return Value
```

### 4.4 Deprecation قبل الحذف

لا يُحذف أي عنصر من API الخارجي مباشرة. مسار الحذف الإلزامي:

```
الإصدار N:    تُضاف الميزة الجديدة
الإصدار N+1:  تُعلَّم الميزة القديمة بـ @deprecated مع رسالة توضح البديل
الإصدار N+2:  تُحذف الميزة القديمة (MAJOR bump إلزامي)
```

الحد الأدنى بين Deprecation والحذف: **إصدار MINOR واحد كامل**.

### 4.5 Database Schema Compatibility

تغييرات قاعدة البيانات تخضع لقواعد إضافية:

```
✅ إضافة عمود nullable جديد          ← آمن، backward-compatible
✅ إضافة جدول جديد                   ← آمن، backward-compatible
✅ إضافة Index                        ← آمن، قد يبطئ الترقية مؤقتاً
⚠️ إضافة عمود NOT NULL بدون default  ← يتطلب Migration دقيقة
❌ حذف عمود                          ← Breaking — MAJOR + Migration
❌ إعادة تسمية عمود أو جدول          ← Breaking — MAJOR + Migration
❌ تغيير نوع بيانات عمود             ← Breaking — MAJOR + Migration
```

## 5. Migration Policy

### 5.1 Migration هي جزء من الإصدار — ليس ملحقاً

عندما يتطلب إصدار ما Migration، يُعدّ ملف Migration جزءاً إلزامياً من هذا الإصدار. لا يُطلَق الإصدار دون Migration كاملة وموثّقة ومُختبَرة.

### 5.2 أنواع Migrations

| النوع | المثال | الخطر | الأسلوب |
|---|---|---|---|
| **Schema Migration** | إضافة عمود / جدول | منخفض | Drizzle migration scripts |
| **Data Migration** | تحويل بيانات موجودة | متوسط | Script + Validation |
| **Destructive Migration** | حذف عمود / جدول | عالٍ | Multi-phase deployment |
| **Session Migration** | تغيير بنية Session | عالٍ | Dual-format support مؤقت |
| **Config Migration** | تغيير أسماء Environment Variables | متوسط | Mapping layer مؤقت |

### 5.3 قاعدة Multi-Phase للـ Destructive Migrations

لا يجوز حذف عمود من قاعدة البيانات في إصدار واحد. المسار الإلزامي:

```
Phase 1 (الإصدار الأول):
  - أضف العمود الجديد
  - اكتب الكود ليقرأ من القديم ويكتب في الاثنين
  - لا تحذف العمود القديم

Phase 2 (الإصدار التالي):
  - اكتب Migration تنسخ البيانات من القديم للجديد
  - اجعل الكود يقرأ من الجديد فقط
  - لا تحذف العمود القديم بعد

Phase 3 (الإصدار الثالث):
  - احذف العمود القديم من Schema
  - احذف أي كود يشير للعمود القديم
```

هذا يضمن إمكانية Rollback في أي مرحلة.

### 5.4 Migration Scripts — متطلبات إلزامية

كل Migration Script يجب أن:

```
1. يكون Idempotent — تشغيله مرتين يُنتج نفس النتيجة
2. يُسجّل بداية تنفيذه ونهايته في اللوغات
3. يتحقق من حالة DB قبل التنفيذ (Pre-condition check)
4. يتحقق من النتيجة بعد التنفيذ (Post-condition check)
5. يتراجع تلقائياً إذا فشلت أي خطوة (Transaction)
6. لا يستغرق أكثر من 30 دقيقة على DB الإنتاج
```

### 5.5 Migration تُختبر أولاً على نسخة من بيانات الإنتاج

قبل تشغيل أي Migration على الإنتاج:
1. أنشئ نسخة من بيانات الإنتاج في بيئة Staging
2. شغّل Migration على النسخة
3. تحقق من سلامة البيانات بعد Migration
4. قِس الوقت الذي استغرقته
5. اختبر Rollback على النسخة

إذا فشلت أي خطوة في Staging — لا تُشغّل Migration في الإنتاج.

### 5.6 توثيق Migration للمستخدم

كل Migration تستلزم إجراءً من المستخدم يجب توثيقها في:
- CHANGELOG بوضوح تحت قسم "Breaking Changes"
- ملف MIGRATION.md خاص بهذا الإصدار مع خطوات تفصيلية
- رسالة تنبيه واضحة قبل الترقية

## 6. Rollback Strategy

### 6.1 Rollback حق — لا ضعف

اتخاذ قرار Rollback ليس اعترافاً بالفشل — هو قرار ناضج يحمي المستخدمين. النظام الذي يمكن التراجع عنه بسرعة أكثر موثوقية من النظام الذي لا يمكن التراجع عنه بثقة.

### 6.2 أنواع Rollback

| النوع | المتى | الأسلوب | الوقت المستهدف |
|---|---|---|---|
| **Immediate Rollback** | خلال 15 دقيقة من الإطلاق | نشر الإصدار السابق | < 5 دقائق |
| **Partial Rollback** | انهيار ميزة بعينها | Feature Flag أو إيقاف Plugin | < 2 دقائق |
| **Database Rollback** | Schema Migration فشلت | تشغيل Down Migration | < 30 دقيقة |
| **Emergency Rollback** | ثغرة أمنية | Immediate + إبطال Sessions | < 10 دقائق |

### 6.3 شروط Rollback الآمن

Rollback آمن فقط إذا:

```
✅ الإصدار السابق لا يزال متاحاً ومُوثَّقاً
✅ Database Schema متوافق مع الإصدار السابق (أو Down Migration موجودة)
✅ لا توجد بيانات كُتبت بصيغة جديدة يعجز الإصدار القديم عن قراءتها
✅ Environment Variables للإصدار القديم موثّقة ومتاحة
✅ اختُبر Rollback في Staging قبل الإصدار
```

### 6.4 متى يُفعَّل Rollback تلقائياً

يُفعَّل Rollback تلقائياً دون انتظار موافقة إذا رُصد أي مما يلي في أول 15 دقيقة بعد الإصدار:

```
🔴 معدل الأخطاء تجاوز 5% (من صفر قبل الإصدار)
🔴 Crash يوقف الـ Process الرئيسي
🔴 Facebook Connection فشل ولم يُستعَد خلال 3 دقائق
🔴 قاعدة البيانات ترفض الاتصال
🔴 لا استجابة من /healthz لأكثر من 60 ثانية
```

### 6.5 توثيق كل Rollback

كل عملية Rollback تُوثَّق فوراً بعد تنفيذها:

```
- الإصدار الذي أُطلق
- الإصدار الذي رُجع إليه
- سبب Rollback (بدقة — ليس "مشكلة في الإصدار")
- وقت اكتشاف المشكلة
- وقت تنفيذ Rollback
- تأثير المستخدمين (عدد / مدة)
- الخطوات التي ستُمنع تكرار هذا
```

لا يُغلق حادث Rollback دون وثيقة Post-Mortem كاملة.

### 6.6 Feature Flags — Rollback الناعم

للميزات ذات الخطر المتوسط، يُستخدم Feature Flag بدلاً من Rollback كامل. يتيح هذا:
- تعطيل الميزة المعطوبة دون التأثير على بقية النظام
- إعادة تفعيلها بعد الإصلاح دون إصدار جديد
- التحكم في نطاق التفعيل (% من المستخدمين)

كل ميزة رئيسية تُطلَق خلف Feature Flag في إصدارها الأول.

## 7. Release Checklist

### 7.1 مراحل الإصدار

```
المرحلة 1: Code Freeze
المرحلة 2: QA (انظر القسم 8)
المرحلة 3: Release Candidate
المرحلة 4: Staging Deployment
المرحلة 5: Production Deployment
المرحلة 6: Post-Release Monitoring
```

### 7.2 Checklist ما قبل Code Freeze

- [ ] جميع المهام المُخطَّط لها في هذا الإصدار مكتملة أو مؤجلة بوعي
- [ ] لا Pull Request مفتوح مُقرَّر دمجه في هذا الإصدار
- [ ] `develop` branch مستقر — لا اختبارات فاشلة
- [ ] رقم الإصدار محدد وفق SemVer
- [ ] CHANGELOG مكتوب ومراجع

### 7.3 Checklist Code Quality

- [ ] `pnpm run typecheck` يمر بصفر أخطاء
- [ ] جميع الاختبارات تنجح
- [ ] لا `console.log` في الكود المُضاف
- [ ] لا TODO بدون Tracking Reference
- [ ] لا Secrets مكتوبة في الكود
- [ ] `pnpm audit` — لا ثغرات Critical أو High
- [ ] Code Review اكتمل لجميع التغييرات

### 7.4 Checklist الوثائق

- [ ] CHANGELOG.md محدَّث
- [ ] الدستور محدَّث إذا أُجري تغيير معماري
- [ ] MIGRATION.md موجود إذا كان هناك Breaking Changes
- [ ] ملاحظات الإصدار كافية للمستخدم لفهم ما الجديد

### 7.5 Checklist قاعدة البيانات

- [ ] جميع Migration Scripts موجودة وصحيحة
- [ ] Migration Scripts اختُبرت على نسخة من بيانات الإنتاج
- [ ] Down Migrations موجودة وصحيحة
- [ ] وقت Migration مقيس ومقبول
- [ ] لا بيانات Orphan ستنتج عن المigration

### 7.6 Checklist الأمان

- [ ] Security Review اكتمل للتغييرات التي تمس Authentication / Session
- [ ] لا Tier 1 Data تُسجَّل في اللوغات
- [ ] جميع المدخلات الجديدة تُتحقق منها بـ Zod
- [ ] Rate Limiting مُطبَّق على الـ Endpoints الجديدة

### 7.7 Checklist Rollback

- [ ] استراتيجية Rollback محددة لهذا الإصدار
- [ ] Rollback اختُبر في Staging
- [ ] الإصدار السابق لا يزال قابلاً للنشر
- [ ] فريق الدعم مُبلَّغ بوقت الإصدار وكيفية التواصل عند المشاكل

### 7.8 Checklist Post-Release

- [ ] `/healthz` يستجيب بعد الإصدار
- [ ] Facebook Connection مُستعاد
- [ ] معدل الأخطاء في المستوى الطبيعي
- [ ] اللوغات لا تُظهر أنماطاً غير متوقعة
- [ ] Monitoring Dashboards تُظهر مؤشرات طبيعية
- [ ] CHANGELOG على GitHub Releases مُحدَّث
- [ ] Tag git مُضاف بالإصدار الجديد

## 8. QA Process

### 8.1 QA ليس خطوة أخيرة — هو ممارسة مستمرة

QA لا يبدأ عند نهاية التطوير — يبدأ من أول سطر كود. الاختبارات تُكتَب مع الكود، لا بعده. كل ميزة جديدة تُرفَق باختباراتها قبل أن تُعدّ مكتملة.

### 8.2 مستويات الاختبار

| المستوى | ما يختبره | متى يُشغَّل |
|---|---|---|
| **Unit Tests** | دالة أو Method واحدة معزولة | عند كل Commit |
| **Integration Tests** | تفاعل طبقتين أو أكثر | عند كل Pull Request |
| **System Tests** | تدفق كامل من Facebook Message للاستجابة | قبل كل Release |
| **Regression Tests** | التأكد أن الموجود لا يزال يعمل | قبل كل Release |
| **Performance Tests** | الأداء تحت الحِمل المتوقع | قبل كل MINOR/MAJOR |
| **Security Tests** | فحص الثغرات الشائعة | قبل كل MINOR/MAJOR |

### 8.3 تغطية الاختبار — الحد الأدنى

| الطبقة | الحد الأدنى للتغطية |
|---|---|
| Core (Types, Interfaces) | N/A — لا منطق تنفيذي |
| Services | 85% — كل Business Logic يُختبَر |
| Commands | 75% — كل Handler مُختبَر |
| Repositories | 70% — كل Query مُختبَرة |
| Middlewares | 80% — كل شرط تحقق مُختبَر |
| Plugins | 70% — كل Handler مُختبَر |
| Utils | 90% — كل دالة مساعدة مُختبَرة |

هذه أرقام **دنيا** — ليست أهدافاً. الهدف دائماً أعلى.

### 8.4 اختبار الحالات الطرفية — إلزامي

لكل ميزة جديدة، تُكتَب اختبارات لـ:

```
✅ الحالة الطبيعية (Happy Path)
✅ المدخلات الفارغة / null / undefined
✅ المدخلات ذات الحجم الأقصى المسموح
✅ تزامن عمليات متعددة على نفس Resource
✅ فشل Dependency خارجي (Facebook API, Database, Cache)
✅ انتهاء صلاحية Session أثناء العملية
✅ إعادة تشغيل الإصدار نفسه (Idempotency)
```

### 8.5 بيئة QA — متطلبات

```
- بيئة Staging مستقلة تماماً عن الإنتاج
- نسخة من بيانات الإنتاج (مُجهَّلة من البيانات الشخصية)
- Facebook Test App معزولة عن حسابات الإنتاج
- نفس إعدادات الإنتاج (Environment Variables) — بقيم اختبار
- Monitoring مُفعَّل في Staging كما هو في الإنتاج
```

### 8.6 Bug Severity Classification

| الدرجة | التعريف | الإجراء |
|---|---|---|
| **P0 — Critical** | النظام لا يعمل / ثغرة أمنية | إيقاف الإصدار فوراً |
| **P1 — High** | ميزة رئيسية معطوبة / فقدان بيانات محتمل | يُصلَح قبل الإصدار |
| **P2 — Medium** | ميزة ثانوية معطوبة / سلوك غير متوقع | يُصلَح أو يُؤجَّل بوعي |
| **P3 — Low** | تحسينات طفيفة / مشاكل جمالية | يُجدوَل في إصدار مستقبلي |

لا يُطلَق إصدار يحمل P0 أو P1 مفتوحاً.

### 8.7 اختبار Rollback في QA

اختبار Rollback جزء لا يتجزأ من QA لكل إصدار:

```
1. نشر الإصدار الجديد في Staging
2. تشغيل System Tests للتحقق من عمله
3. تنفيذ Rollback للإصدار السابق
4. التحقق من أن الإصدار السابق يعمل بشكل صحيح
5. التحقق من سلامة البيانات بعد Rollback
```

إذا فشل اختبار Rollback في Staging — لا يُطلَق الإصدار.

## 9. Deployment Strategy

### 9.1 ترتيب بيئات النشر — لا تجاوز

```
Development (محلي) → Staging → Production

لا يذهب أي كود لـ Production دون المرور بـ Staging أولاً.
لا استثناءات. حتى الـ Hotfixes تُختبَر في Staging.
```

### 9.2 نوافذ النشر

| نوع الإصدار | النافذة المفضّلة | النافذة المحظورة |
|---|---|---|
| **MAJOR** | بداية الأسبوع، صباحاً | الجمعة / عطل نهاية الأسبوع |
| **MINOR** | أي يوم عمل، صباحاً | المساء المتأخر |
| **PATCH** | أي يوم عمل | ليس مقيّداً |
| **Hotfix** | في أي وقت عند الضرورة | لا قيود — الأمان أولاً |

### 9.3 خطوات Deployment في الإنتاج

```
الخطوة 1: إشعار (5 دقائق قبل)
  - تسجيل وقت بدء النشر
  - التأكد من وجود شخص مسؤول يراقب أثناء النشر

الخطوة 2: Database Migrations
  - تشغيل Migration Scripts إذا وُجدت
  - التحقق من نجاح Migration قبل المتابعة

الخطوة 3: نشر الكود الجديد
  - استخدام Zero-Downtime Deployment إذا أمكن
  - Graceful Shutdown للعمليات الجارية

الخطوة 4: التحقق الفوري (أول 5 دقائق)
  - /healthz يستجيب ✅
  - Facebook Connection مُستعاد ✅
  - لا أخطاء غير متوقعة في اللوغات ✅

الخطوة 5: المراقبة الموسّعة (أول 30 دقيقة)
  - معدل الأخطاء في المستوى الطبيعي
  - Latency في النطاق المتوقع
  - لا تدهور في أداء قاعدة البيانات

الخطوة 6: الإعلان عن نجاح الإصدار
  - تحديث GitHub Releases
  - إضافة Git Tag
```

### 9.4 Zero-Downtime Deployment

Void يستهدف نشراً بدون توقف. المتطلبات:

```
- Database Migrations يجب أن تكون backward-compatible أثناء النشر
- الإصدار القديم والجديد يجب أن يعملا جنباً إلى جنب مؤقتاً
- Session Format يجب أن يُقرأ من كلا الإصدارين أثناء الانتقال
- الـ Health Check يجب أن يفشل بوضوح إذا الـ Service غير جاهزة
```

### 9.5 Feature Flags كأداة نشر

الميزات الكبيرة تُنشَر خلف Feature Flags بشكل إلزامي:

```
المرحلة 1: نشر الكود مع الـ Flag = false (لا تأثير على المستخدمين)
المرحلة 2: تفعيل الـ Flag لـ 1% من المستخدمين — مراقبة
المرحلة 3: تفعيل الـ Flag لـ 10% — مراقبة
المرحلة 4: تفعيل الـ Flag لـ 50% — مراقبة
المرحلة 5: تفعيل الـ Flag لـ 100%
المرحلة 6: حذف الـ Flag من الكود في الإصدار التالي
```

### 9.6 Hotfix Deployment — المسار الطارئ

الـ Hotfix مسار مُسرَّع لإصلاحات P0 فقط:

```
1. إنشاء فرع hotfix/* من main (ليس من develop)
2. تطبيق الإصلاح المحدد فقط — لا إضافات جانبية
3. اختبار مُركَّز في Staging (30 دقيقة كحد أدنى)
4. نشر في الإنتاج مع مراقبة مكثّفة
5. دمج hotfix في develop و main كلاهما
6. إصدار PATCH فوري موثَّق
```

## 10. AI Release Rules

### 10.1 AI لا يُطلق إصدارات بشكل مستقل

لا يملك أي AI صلاحية إطلاق إصدار في الإنتاج بدون موافقة صريحة من المستخدم المسؤول. AI يُحضّر، يتحقق، يُوثّق — لكن القرار النهائي للإنسان.

### 10.2 AI يُعيَّن رقم الإصدار الصحيح

عند مشاركة AI في تحضير إصدار، يجب أن:

```
1. يُراجع جميع التغييرات منذ آخر إصدار
2. يُصنّف كل تغيير: Breaking / New Feature / Bug Fix
3. يُحدد رقم الإصدار الصحيح وفق SemVer
4. يُبرر اختياره بأمثلة من التغييرات
5. لا يُخفّض الرقم لأنه "يبدو كبيراً" — الدقة قبل الراحة
```

**مثال على تبرير صحيح:**
```
التغييرات تشمل حذف Method من IUserRepository
هذا Breaking Change → يجب رفع MAJOR
الإصدار الجديد: 2.0.0 (كان 1.4.2)
```

### 10.3 AI يكتب CHANGELOG بدقة

CHANGELOG الذي يكتبه AI يجب أن:

```
✅ يصف ما تغيّر فعلياً — لا ما كان مخططاً
✅ يُميّز بوضوح بين Breaking Changes وNew Features وBug Fixes
✅ يذكر كل Breaking Change مع مثال على كيفية التكيّف معه
✅ يكون بلغة واضحة يفهمها المستخدم النهائي
✅ يستخدم أفعالاً محددة: "أضاف، حذف، أصلح، غيّر، أزال"
```

**تنسيق CHANGELOG الإلزامي:**

```markdown
## [2.1.0] - 2026-07-03

### Breaking Changes
- **IUserRepository**: حُذفت `findByPhoneNumber()` — استخدم `findByIdentifier({ type: 'phone', value })` بدلاً منها
- **Session Format**: تغيّر هيكل SessionData — يتطلب Migration

### New Features
- أُضيف `OrderCommand` لمعالجة طلبات الشراء
- أُضيف Event Type جديد: `order.created`

### Bug Fixes
- أُصلح Memory Leak في EventDispatcher عند تسجيل وإلغاء Subscriptions بشكل متكرر
- أُصلح Race Condition في SessionManager عند تجديد Session تحت حِمل عالٍ

### Performance
- تحسّن وقت استجابة UserService.getOrCreate بنسبة 40%
```

### 10.4 AI يتحقق من Checklist قبل إعلان الجاهزية

قبل أن يُعلن AI أن إصداراً جاهز للمراجعة، يُشغّل بنفسه:

```
- [ ] typecheck يمر بصفر أخطاء
- [ ] جميع الاختبارات تنجح
- [ ] لا console.log في الكود الجديد
- [ ] لا Secrets في الكود
- [ ] CHANGELOG محدَّث
- [ ] رقم الإصدار صحيح وفق SemVer
- [ ] Migration موجودة إذا لزم
- [ ] Breaking Changes موثّقة
```

إذا فشل أي بند — AI يُصلحه قبل إعلان الجاهزية، لا يُرجّئه.

### 10.5 AI لا يُسهم في ثقافة "الإصدار السريع"

إذا طُلب من AI تجاوز خطوة في عملية الإصدار بحجة ضغط الوقت:

```
1. يرفض التجاوز بوضوح
2. يُقدّر الوقت اللازم لإتمام الخطوة بشكل صحيح
3. يقترح ما يمكن تأجيله للإصدار القادم بدلاً من تجاوز ضمانات الجودة
4. يُذكّر بتكلفة الإصدار المعطوب مقارنةً بتأخير ساعات
```

### 10.6 AI يُوثّق قرارات الإصدار

أي قرار غير اعتيادي اتُّخذ أثناء تحضير الإصدار — تأجيل ميزة، تغيير رقم الإصدار، اختيار نافذة نشر خاصة — يُوثّقه AI في ملاحظات الإصدار مع السبب. لا قرارات صامتة.

### 10.7 AI يُراقب ما بعد الإصدار

بعد كل إصدار، يتابع AI لمدة 30 دقيقة:

```
- معدل الأخطاء في اللوغات
- استجابة /healthz
- حالة Facebook Connection
- أي نمط غير اعتيادي في Metrics
```

إذا رصد أي مؤشر P0 أو P1 — يُبلّغ المستخدم فوراً ويُوصي بـ Rollback دون انتظار.
