# 25 — AI Development Rules

> **Status:** Official
> **Scope:** كل نظام ذكاء اصطناعي يقرأ هذا الكود أو يكتبه أو يعدّله — بلا استثناء
> **Authority:** هذا الملف هو المرجع الرسمي الوحيد الذي يحكم سلوك أي AI يعمل على مشروع Void. لا يجوز لأي AI البدء في أي عمل قبل قراءة هذا الملف كاملاً وفهمه. أي كود يخالف القواعد المحددة هنا هو كود مرفوض — بصرف النظر عن صحته الوظيفية.

## 1. قبل كل شيء — قراءة الدستور

### 1.1 الدستور يُقرأ أولاً — دائماً

لا يجوز لأي AI كتابة سطر كود واحد في Void قبل قراءة ملفات الدستور ذات الصلة بالمهمة المطلوبة. هذه ليست توصية — هي شرط غير قابل للتجاوز.

**ترتيب القراءة الإلزامي عند بدء أي مهمة:**

```
1. هذا الملف (25-ai-development-rules.md)          ← أول ما يُقرأ دائماً
2. 00-project-specification.md                      ← ما هو المشروع وما هدفه
3. 02-architecture.md                               ← الهيكل العام والطبقات
4. 03-layer-responsibilities.md                     ← مسؤولية كل طبقة
5. 04-dependency-rules.md                           ← من يعتمد على من
6. الملف المتخصص في الطبقة المعنية بالمهمة        ← حسب الحاجة
```

**مثال:** إذا كانت المهمة تتعلق بإضافة Command جديد، يُقرأ الترتيب أعلاه ثم `13-command-system.md`. إذا كانت تتعلق بـ Session، يُقرأ `11-session-management.md`. لا اختصارات.

### 1.2 الدستور هو مصدر الحقيقة

عندما يتعارض ما يطلبه المستخدم مع ما يقرره الدستور، **الدستور يكسب**. ليس لأن المستخدم مخطئ، بل لأن الدستور يحمي تماسك المشروع على المدى البعيد. على AI أن يشرح التعارض ويقترح البديل المتوافق — لا أن يتجاهل الدستور خدمةً للطلب.

### 1.3 الجهل ليس عذراً

إذا لم يقرأ AI ملف دستور ذا صلة بالمهمة وكتب كوداً يخالفه، فالكود مرفوض. "لم أكن أعرف" ليست مبرراً مقبولاً. الدستور موجود ومتاح — على AI قراءته.

### 1.4 فهم المشروع قبل تعديله

لا يجوز لـ AI افتراض بنية المشروع — يجب التحقق منها. قبل إضافة أي شيء:

```
✅ اقرأ الملفات القائمة في الطبقة المعنية
✅ افهم الأنماط المستخدمة فيها
✅ تحقق من وجود Interface مناسب في Core
✅ تحقق من وجود Service أو Repository موجود يخدم الغرض
```

## 2. كيف يقرأ AI المشروع

### 2.1 خريطة الاستكشاف

عند مواجهة المشروع لأول مرة أو بعد غياب، يتبع AI هذا الترتيب:

**المرحلة الأولى — الفهم المعماري (لا كود بعد):**
```
1. قراءة .constitution/README.md                   ← نظرة عامة على الدستور
2. قراءة 00-project-specification.md              ← ما الذي يفعله Void
3. قراءة 02-architecture.md                        ← الطبقات وعلاقاتها
4. قراءة 03-layer-responsibilities.md              ← مسؤولية كل طبقة
```

**المرحلة الثانية — الفهم التقني:**
```
5. مراجعة هيكل المجلدات (بدون قراءة التفاصيل بعد)
6. قراءة Core Layer — الـ Interfaces والـ Types
7. قراءة ملف الـ DI Container لفهم كيف تُحقَن التبعيات
8. قراءة نقطة الدخول الرئيسية (index.ts أو main.ts)
```

**المرحلة الثالثة — الفهم المحلي (قبل أي تعديل):**
```
9. قراءة الملفات المتعلقة مباشرةً بالمهمة المطلوبة
10. قراءة الاختبارات المصاحبة لهذه الملفات
11. فهم كيف يُستخدَم هذا المكون من بقية النظام
```

### 2.2 لا افتراضات — تحقق دائماً

| بدلاً من افتراض... | افعل... |
|---|---|
| "يجب أن يكون هناك `UserService`" | ابحث عنه في Core Interfaces أولاً |
| "هذا المكون يستخدم PostgreSQL" | تحقق من `db-config.ts` أو `drizzle.config.ts` |
| "هذا Event موجود" | ابحث في `event-types.ts` |
| "هذا Pattern مدعوم" | تحقق من `13-command-system.md` |
| "هذا Interface يقبل هذا النوع" | اقرأ تعريف Interface في Core |

### 2.3 قراءة الاختبارات كوثيقة

اختبارات المشروع هي وثيقة حية تصف السلوك المقصود. قبل تعديل أي مكون، يقرأ AI اختباراته ليفهم:
- ما الذي يُعدّ سلوكاً صحيحاً
- ما الحالات الطرفية التي كان يجب معالجتها
- ما الافتراضات التي يعتمد عليها

### 2.4 قراءة Git History عند الحيرة

إذا لم يكن سبب وجود كود ما واضحاً، تاريخ Git يحمل الإجابة. تغيير غير مفهوم غالباً له سبب موثّق في commit message. لا يُحذف كود "غريب المظهر" دون فهم سبب وجوده.

## 3. كيف يقرر AI مكان الكود

### 3.1 دليل القرار — اطرح هذه الأسئلة بالترتيب

قبل كتابة أي سطر كود، يمر AI بهذا المسار الإلزامي:

```
السؤال 1: هل هذا عقد (Interface) أو نوع (Type) أو خطأ مخصص؟
  → نعم: Core Layer — لا نقاش.

السؤال 2: هل هذا يتعلق بالتواصل مع Facebook API؟
  → نعم: Facebook Layer — لا نقاش.

السؤال 3: هل هذا يحتوي قراراً أو قاعدة أعمال؟
  → نعم: Services Layer — لا نقاش.

السؤال 4: هل هذا يربط نمط رسالة بـ Handler ويُفوّض للـ Service؟
  → نعم: Commands Layer — لا نقاش.

السؤال 5: هل هذا وظيفة اختيارية مستقلة قابلة للتفعيل والتعطيل؟
  → نعم: Plugins Layer — لا نقاش.

السؤال 6: هل هذا يُطبَّق أفقياً على جميع الطلبات؟
  → نعم: Middleware Layer — لا نقاش.

السؤال 7: هل هذا يُدير دورة حياة مورد دون business logic؟
  → نعم: Managers Layer — لا نقاش.

السؤال 8: هل هذا يخزن أو يسترجع البيانات من قاعدة البيانات؟
  → نعم: Database Layer (Repository) — لا نقاش.

السؤال 9: هل هذا يخزن أو يسترجع بيانات مؤقتة؟
  → نعم: Cache Layer — لا نقاش.

السؤال 10: هل هذا مهمة دورية لا تحتاج تدخل المستخدم؟
  → نعم: Scheduler Layer — لا نقاش.

السؤال 11: هل هذا دالة مساعدة عامة لا تعرف شيئاً عن النطاق؟
  → نعم: Utils Layer — لا نقاش.

السؤال 12: هل هذا يقرأ إعدادات من Environment Variables؟
  → نعم: Configs Layer — لا نقاش.

لم تُجب بـ "نعم" على أي سؤال؟
  → توقف. ناقش مع المستخدم قبل المتابعة.
```

### 3.2 الكود له مكان واحد فقط

لا يجوز وضع نفس المنطق في أكثر من طبقة. إذا وجد AI نفسه يكتب business logic في Command Handler لأنه "أسهل"، هذا خطأ معماري — المنطق ينتمي للـ Service بغض النظر عن السهولة.

### 3.3 اسم الملف يعكس طبقته

```
✅ UserService.ts          → Services Layer
✅ UserRepository.ts       → Database Layer
✅ StartCommand.ts         → Commands Layer
✅ AnalyticsPlugin.ts      → Plugins Layer
✅ IUserService.ts         → Core Layer
✅ RateLimiterMiddleware.ts → Middleware Layer
✅ SessionManager.ts       → Managers Layer

❌ UserHelper.ts           → اسم غامض — ما الطبقة؟
❌ Utils.ts                → حقيبة متنوعة — غير مسموح
❌ Misc.ts                 → مرفوض بالكامل
```

## 4. متى يرفض AI تعديل المعمارية

### 4.1 الرفض ليس عصياناً — هو حماية

AI الذي يرفض طلباً معمارياً خاطئاً لا يعصي المستخدم — يحميه. كود يخالف معمارية Void سيكلف أكثر بكثير لاحقاً مما يوفره الآن.

### 4.2 حالات الرفض الإلزامي

يرفض AI تنفيذ أي طلب يؤدي إلى:

**انتهاكات Layer Boundaries:**
- Business Logic خارج Services
- Facebook API داخل Services مباشرة
- Database queries خارج Repositories
- Plugin تستدعي Plugin أخرى مباشرة
- Service تعرف اسم Command استدعاها

**انتهاكات Dependency Rules:**
- طبقة عليا تعتمد على طبقة دنيا (مثل Core تستورد من Services)
- Circular Dependencies بين Services
- Plugin تعدّل Interface في Core

**انتهاكات نظام الأحداث:**
- Handler يُستدعى مباشرة دون المرور بـ EventDispatcher
- مكون يشترك في حدث غير موجود في الكتالوج
- Plugin تلغي Core Events

**انتهاكات الأمان:**
- Secrets مكتوبة مباشرة في الكود
- تجاوز Input Validation
- تسجيل بيانات Tier 1 في اللوغات

### 4.3 كيف يرفض AI بشكل صحيح

الرفض يتبع هذا النمط دائماً:

```
1. اذكر الطلب بوضوح: "طُلب مني [كذا]"
2. اذكر القاعدة المنتهكة: "هذا يخالف [المبدأ] في [الملف]"
3. اشرح لماذا القاعدة موجودة: "السبب هو [كذا]"
4. قدّم البديل الصحيح: "الطريقة الصحيحة هي [كذا]"
```

**مثال:**
> "طُلب مني وضع query مباشرة في `UserService`. هذا يخالف قاعدة 'لا Database داخل Services' من `03-layer-responsibilities.md`. القاعدة موجودة لعزل منطق التخزين وجعل Services قابلة للاختبار دون قاعدة بيانات. الطريقة الصحيحة: أنشئ `UserRepository` يُطبّق `IUserRepository` من Core، وأحقن الـ Interface في Service."

### 4.4 متى يُعاد النظر في المعمارية

التعديل المعماري ليس مستحيلاً — لكنه يسلك طريقاً محدداً:

```
1. توثيق سبب التغيير في ADR (Architectural Decision Record)
2. تحديث الدستور أولاً قبل تغيير الكود
3. الحصول على موافقة صريحة من المستخدم
4. تطبيق التغيير بشكل متسق في جميع أنحاء القاعدة
```

AI لا يُعدّل المعمارية بصمت. إذا رأى أن تغييراً معمارياً ضروري، يطرحه للنقاش أولاً.

## 5. متى ينشئ AI ملفاً جديداً

### 5.1 شروط إنشاء ملف جديد

يُنشأ ملف جديد **فقط** عند توافر **جميع** هذه الشروط:

```
الشرط 1: المسؤولية التي سيحملها الملف واضحة وقابلة للتعبير عنها بجملة واحدة.
الشرط 2: هذه المسؤولية لا يملكها ملف موجود بالفعل.
الشرط 3: الملف الجديد ينتمي لطبقة محددة في المعمارية.
الشرط 4: اسم الملف يعبّر بدقة عن مسؤوليته ويتبع اتفاقيات التسمية.
الشرط 5: لا يمكن تحقيق نفس الهدف بتوسيع ملف موجود.
```

### 5.2 حالات إنشاء ملف جديد — أمثلة

```
✅ إنشاء OrderService.ts   ← مسؤولية جديدة واضحة: إدارة الطلبات
✅ إنشاء OrderRepository.ts ← مستودع جديد لجدول جديد في DB
✅ إنشاء CheckoutCommand.ts ← Command جديد لنمط رسالة جديد
✅ إنشاء IOrderService.ts   ← Interface جديد في Core لخدمة جديدة
✅ إنشاء PaymentPlugin.ts   ← Plugin مستقلة جديدة بوظيفة محددة
```

### 5.3 حالات لا تستدعي ملفاً جديداً

```
❌ "أريد تنظيم الكود أحسن" → أعد هيكلة الموجود، لا تضف ملفات
❌ "هذا Method طويل" → استخرج helper functions في نفس الملف
❌ "أريد تجميع كل Constants" → Constants تنتمي لملفات طبقاتها
❌ "سنحتاج هذا مستقبلاً" → لا تنشئ ملفات لمتطلبات افتراضية
```

### 5.4 ترتيب إنشاء الملفات

عند إضافة قدرة جديدة، يُتبع هذا الترتيب الإلزامي:

```
1. Interface في Core أولاً         ← يحدد العقد
2. Types والـ Errors في Core       ← يحدد البنية
3. Repository (إن كان هناك DB)    ← التخزين
4. Service                         ← المنطق
5. Command أو Plugin (إن لزم)     ← نقطة الدخول
6. Middleware (إن لزم)            ← المعالجة الأفقية
7. Tests لكل طبقة                ← التحقق
```

هذا الترتيب ليس اختيارياً. Core يأتي أولاً لأن بقية الطبقات تعتمد عليه.

## 6. متى يعيد AI استخدام كود موجود

### 6.1 إعادة الاستخدام أولاً — الإنشاء أخيراً

قبل كتابة أي كود، يبحث AI عمّا هو موجود بالفعل. الكود المكرر هو عدو المشروع — يضاعف موضع الأخطاء ويجعل التغييرات مؤلمة.

**سلوك البحث الإلزامي:**
```
1. ابحث في Core عن Interface يخدم الغرض
2. ابحث في Services عن منطق مشابه قائم
3. ابحث في Utils عن دوال مساعدة ذات صلة
4. ابحث في Repositories عن استعلامات مشابهة
```

### 6.2 متى تُعاد الاستخدام — أمثلة

```
✅ Service موجود لها Method تخدم الغرض → استدعها، لا تكررها
✅ Repository لها Query تخدم الغرض → استخدمها مع Parameters مختلفة
✅ Middleware موجودة تؤدي التحقق المطلوب → أضفها للـ Pipeline، لا تُعد كتابتها
✅ Utility function في Utils → استوردها، لا تُعد كتابتها
✅ Event Type موجود في الكتالوج → استخدمه، لا تنشئ نوعاً مشابهاً
```

### 6.3 متى لا يصح إعادة الاستخدام

```
❌ Method تفعل شيئاً مشابهاً لكن مختلفاً في التفاصيل → لا تُشوّه الموجود، أنشئ Method جديدة
❌ Service لطبقة مختلفة → لا تستخدم Service من طبقة غير مصرح بها
❌ الكود الموجود يحتاج تعديلاً جوهرياً ليخدم الغرض الجديد → ناقش الأمر مع المستخدم أولاً
```

### 6.4 التوسع مقبول — الانتهاك ليس كذلك

إضافة Method جديدة لـ Service قائمة مقبول إذا كانت المسؤولية الجديدة ضمن نطاق تلك الـ Service. لكن توسيع Service لتفعل شيئاً خارج نطاق مسؤوليتها — لأن ذلك "أسرع" — انتهاك للـ Single Responsibility Principle.

## 7. متى يسأل AI المستخدم

### 7.1 السؤال واجب — لا اختياري

في الحالات التالية، يتوقف AI عن العمل ويسأل قبل المتابعة. المضي دون سؤال هو خطأ.

### 7.2 حالات السؤال الإلزامي

**الغموض المعماري:**
- الطلب يمكن تنفيذه بأكثر من طريقة لكل منها مقايضة مختلفة
- المهمة تتعارض مع قاعدة في الدستور ولا يوجد بديل واضح
- المهمة تؤثر على أكثر من طبقة بطريقة غير مضمونة العواقب

**الطلبات ذات الأثر الواسع:**
- تعديل Interface في Core (يؤثر على كل المُطبِّقين)
- تغيير Event Type في الكتالوج (يؤثر على كل المشتركين)
- حذف مكون أو طبقة كاملة
- تغيير استراتيجية Authentication أو Session

**حين يكتشف AI مشكلة لم تُذكر:**
- اكتشاف ثغرة أمنية أثناء التنفيذ
- اكتشاف أن الميزة المطلوبة تكسر ميزة أخرى قائمة
- اكتشاف أن افتراضاً في الطلب غير صحيح

**حين تكون المتطلبات ناقصة:**
- الطلب لا يحدد أي Event يجب إطلاقه عند حدث معين
- الطلب لا يحدد سلوك حالات الفشل
- الطلب يفترض بنيانة غير موجودة

### 7.3 حالات لا تستدعي سؤالاً

```
❌ "كيف يجب أن أسمي هذا المتغير؟" → قرر وفق اتفاقيات التسمية
❌ "أي Library أستخدم؟" → اختر الأنسب ووثّق السبب
❌ "هل أضيف Logging هنا؟" → نعم دائماً — اتبع 08-logging-policy.md
❌ "هل أكتب الاختبارات؟" → نعم دائماً — اتبع 21-testing-policy.md
```

### 7.4 كيف يسأل AI بشكل صحيح

السؤال الجيد يكون:
```
1. محدداً: "أحتاج توضيحاً حول [نقطة محددة]"
2. مع الخيارات: "يمكنني فعل [أ] أو [ب] — لكل منهما عواقب [كذا]"
3. مع توضيح الأثر: "هذا القرار سيؤثر على [المكونات التالية]"
4. مع توصية: "أنصح بـ [الخيار] لأن [السبب]"
```

## 8. كيف يجري AI عملية Refactoring

### 8.1 Refactoring يحسّن الشكل دون تغيير السلوك

التعريف الدقيق: Refactoring يُغيّر هيكل الكود دون تغيير سلوكه الخارجي. إذا تغير السلوك، هذا ليس Refactoring — هذا تعديل وظيفي يخضع لقواعد مختلفة.

### 8.2 شروط Refactoring آمن

```
الشرط 1: الاختبارات موجودة وتغطي السلوك المراد إعادة هيكلته
الشرط 2: الاختبارات تمر قبل البدء وبعد الانتهاء
الشرط 3: التغيير يتم خطوة واحدة في كل مرة (لا Big Bang)
الشرط 4: كل خطوة يمكن التراجع عنها بشكل مستقل
```

### 8.3 ترتيب Refactoring الإلزامي

```
الخطوة 1: تشغيل الاختبارات وتوثيق نتيجتها الحالية
الخطوة 2: إجراء تغيير صغير واحد
الخطوة 3: تشغيل الاختبارات للتأكد من استمرار نجاحها
الخطوة 4: تكرار حتى الانتهاء
```

لا يجمع AI تغييرات متعددة في خطوة واحدة. Refactoring يُجرى بشكل تدريجي.

### 8.4 أنواع Refactoring المسموح بها

```
✅ استخراج Method من method طويلة (Extract Method)
✅ تسمية عناصر بأسماء أوضح (Rename)
✅ نقل مسؤولية من طبقة إلى طبقتها الصحيحة
✅ إزالة تكرار (DRY)
✅ تبسيط Conditionals المعقدة
✅ استخراج Interface من Implementation موجودة
```

### 8.5 Refactoring المحظور

```
❌ تغيير سلوك مكون وتسميته Refactoring
❌ Refactoring مكون بدون اختبارات تغطيه
❌ تغيير واجهة Public دون تحديث جميع المُستخدِمين
❌ دمج طبقتين في ملف واحد "لأنهما صغيرتان"
❌ نقل Business Logic للـ Middleware "لأنه أسرع"
```

### 8.6 توثيق Refactoring

كل Refactoring يرافقه:
- شرح **لماذا** أُجري (ليس ماذا فعل)
- توثيق أي افتراضات تتغير
- تحديث الدستور إذا أفرز الـ Refactoring قاعدة عامة جديدة

## 9. كيف يحافظ AI على استقرار المشروع

### 9.1 الاستقرار ليس خاصية — هو حق

المشروع الذي يعمل عليه AI هو مشروع حي يخدم مستخدمين حقيقيين. كل تغيير يحتمل كسر شيء. المحافظة على الاستقرار ليست اعتباراً ثانوياً — هي مسؤولية أولى.

### 9.2 مبدأ "أصغر تغيير ممكن"

عند حل مشكلة، يختار AI أصغر تغيير يحل المشكلة — لا أكثر. التغييرات الكبيرة تحمل مخاطر كبيرة. إذا كان حل المشكلة يتطلب تغييراً واسعاً، يُقسَّم إلى مراحل صغيرة.

```
❌ "سأُعيد كتابة كامل نظام الـ Session لأنني رأيت طريقة أفضل"
✅ "سأُصلح هذا Bug المحدد، وأوثّق رأيي في تحسين النظام للمناقشة لاحقاً"
```

### 9.3 قائمة التحقق قبل كل تسليم

قبل اعتبار أي مهمة منجزة، يُجيب AI على هذه الأسئلة:

**الصحة الوظيفية:**
- [ ] هل الكود الجديد يفعل ما طُلب منه؟
- [ ] هل جميع حالات الفشل مُعالَجة؟
- [ ] هل الكود يعمل تحت الحِمل الطبيعي والاستثنائي؟

**التوافق المعماري:**
- [ ] هل الكود في طبقته الصحيحة؟
- [ ] هل التبعيات تسير في الاتجاه الصحيح؟
- [ ] هل Interface موجود في Core إذا كان المكون يُستخدَم من طبقات أخرى؟

**الأمان:**
- [ ] هل جميع المدخلات تُتحقق منها؟
- [ ] هل لا توجد بيانات حساسة في اللوغات؟
- [ ] هل الكود يتبع `09-security-policy.md`؟

**الاختبار:**
- [ ] هل الاختبارات موجودة وتغطي السلوك الجديد؟
- [ ] هل الاختبارات القائمة لا تزال تنجح؟

**التوثيق:**
- [ ] هل الـ Interfaces الجديدة مُوثَّقة؟
- [ ] هل الـ Why-comments موجودة للكود غير البديهي؟
- [ ] هل الدستور بحاجة للتحديث؟

### 9.4 التراجع عن التغييرات

إذا أدى تغيير إلى كسر شيء لم يكن متوقعاً، AI لا يُضيف كوداً فوق كود معطوب — يتراجع أولاً، يفهم المشكلة، ثم يُحل بشكل صحيح. حل المشكلة بطبقة إضافية من الكود هو مرض طويل الأمد.

### 9.5 الأنظمة الحرجة تستدعي حذراً مضاعفاً

بعض المكونات في Void ذات طابع حرج — أي خطأ فيها يؤثر على كامل النظام:

| المكون | سبب الحساسية |
|---|---|
| Facebook Session | خسارته = انقطاع كامل عن المنصة |
| EventDispatcher | تعطله = توقف كامل نظام الأحداث |
| AuthMiddleware | خطأ فيه = ثغرة أمنية |
| DI Container | خطأ في التهيئة = انهيار كامل عند Startup |
| Config Loader | قيمة خاطئة = سلوك غير متوقع في الإنتاج |

أي تعديل على هذه المكونات يستوجب:
- مراجعة مضاعفة
- اختبارات تغطية شاملة
- إشعار صريح للمستخدم بطبيعة التغيير وأثره

## 10. المحظورات المطلقة

هذه محظورات غير قابلة للتجاوز تحت أي ظرف أو بأي مبرر.

### 10.1 محظورات الكود

```
❌ كتابة Secrets أو API Keys مباشرة في الكود
❌ تعطيل Input Validation تحت أي مسمى
❌ استدعاء Handler مباشرة دون EventDispatcher
❌ وضع SQL مباشرة في Service أو Command
❌ استخدام console.log — استخدم logger دائماً
❌ إنشاء Circular Dependencies بين Services
❌ تعديل Core Interfaces من داخل Plugin
❌ تجاوز Repository للوصول المباشر للـ DB من Service
❌ إرسال رسالة Facebook مباشرة من Service
```

### 10.2 محظورات الأسلوب

```
❌ البدء في الكود قبل قراءة الدستور ذي الصلة
❌ إنشاء "طبقة جديدة" غير موجودة في الدستور دون نقاش
❌ تجاهل الاختبارات الفاشلة والمتابعة رغمها
❌ كتابة "سنُحسّن هذا لاحقاً" دون Tracking واضح
❌ تغيير المعمارية بصمت دون توثيق أو إشعار
❌ نسخ كود من طبقة إلى أخرى بدلاً من بناء الـ Interface الصحيح
```

### 10.3 محظورات القرار

```
❌ اتخاذ قرار معماري جوهري دون سؤال المستخدم
❌ رفض مبدأ في الدستور لأن "الطريقة الأخرى أسرع"
❌ افتراض أن قاعدة لا تنطبق على حالة بعينها دون مراجعة
❌ تطبيق Refactoring واسع دون موافقة مسبقة
```

## 11. قائمة المراجعة الذاتية

يستخدم AI هذه القائمة على نفسه قبل تسليم أي مهمة.

### قبل البدء
- [ ] قرأت هذا الملف كاملاً
- [ ] قرأت ملفات الدستور ذات الصلة بالمهمة
- [ ] فهمت بنية المشروع القائمة قبل إضافة أي شيء
- [ ] لم أبدأ في الكتابة قبل تحديد الطبقة الصحيحة

### أثناء التنفيذ
- [ ] الكود في طبقته الصحيحة وفق دليل القرار في القسم 3
- [ ] التبعيات تسير في الاتجاه الصحيح (الأعلى يعتمد على الأدنى، Core لا يعتمد على أحد)
- [ ] لا business logic خارج Services
- [ ] لا Database access خارج Repositories
- [ ] لا Facebook API calls خارج Facebook Layer
- [ ] لا Secrets في الكود
- [ ] جميع المدخلات الخارجية تُتحقق منها
- [ ] جميع الأحداث تمر عبر EventDispatcher
- [ ] Logger مستخدم بدلاً من console.log

### قبل التسليم
- [ ] الاختبارات موجودة وتنجح
- [ ] الاختبارات القائمة لا تزال تنجح
- [ ] الـ Interfaces الجديدة مُوثَّقة
- [ ] الكود غير البديهي له Why-comment
- [ ] لم أتجاوز حدود أي طبقة
- [ ] إذا أجريت Refactoring: السلوك لم يتغير
- [ ] إذا اكتشفت مشكلة لم تُذكر: أبلغت المستخدم
- [ ] الدستور لا يحتاج تحديثاً بسبب هذا التغيير — أو حدّثته

---

*هذا الملف هو المرجع الرسمي والوحيد الذي يحكم سلوك أي ذكاء اصطناعي يعمل على مشروع Void. لا يجوز تجاوز أي قاعدة فيه بأي مبرر. القواعد هنا ليست اقتراحات — هي الحد الأدنى للكفاءة المقبولة. AI يخالف هذه القواعد يُنتج كوداً يضر بالمشروع أكثر مما ينفعه. كل مخالفة مكتشفة يجب الإبلاغ عنها وتصحيحها فوراً — لا التغاضي عنها.*
